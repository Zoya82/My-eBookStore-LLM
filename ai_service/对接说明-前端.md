# 智能服务对接说明（给前端）

前端只需要**调用三个接口**，不用关心里面怎么调大模型。所有接口都是 `POST`，收发都是 JSON。

## 一、服务地址

前端配一个 `baseURL` 变量统一管理：

```js
// config.js
export const AI_BASE = 'http://localhost:8001'  // 联调期指向跑 AI 服务的机器；集成后换后端地址
```

- **联调期**：指向运行 AI 服务的机器（本机 `localhost`，跨机用对方局域网 IP）
- **集成后**：挂进后端，用统一后端地址，前缀 `/api/ai` 不变

## 二、三个接口

### 1. 智能摘要 `POST /api/ai/summary`
**用途**：把一段长的图书简介，压缩成一句话。用在**图书详情页**（简介太长时给个速读版）。

请求：
```json
{ "text": "要摘要的图书简介（必填）", "max_length": 60 }
```
返回：
```json
{ "success": true, "summary": "一句话摘要", "message": "" }
```

### 2. 智能推荐 `POST /api/ai/recommend`
**用途**：用户用自然语言说想看什么，返回推荐书单。用在**搜索/推荐页**或**大模型对话页**。

请求：
```json
{ "query": "想找一本适合周末放松、治愈系的书" }
```
返回：
```json
{
  "success": true,
  "items": [ { "id": 6, "title": "小王子", "reason": "纯净温暖，治愈心灵" } ],
  "reply": "模型原始文本（一般不用，兜底展示时才用）",
  "message": ""
}
```
> `items` 是结构化推荐结果，**优先用它渲染**（每项有书 id / 书名 / 推荐理由）。

### 3. 多轮问答 `POST /api/ai/chat`
**用途**：图书客服对话，能记住上下文。用在**大模型对话页**。上下文由**后端按 `session_id` 自动维护**（30 分钟无活动自动过期），前端不用管历史。

请求：
```json
{ "message": "用户本轮说的话", "session_id": "首次不传，之后带上一次返回的" }
```
返回：
```json
{ "success": true, "reply": "助手的回答", "session_id": "abc123", "message": "" }
```

## 三、统一的成功/失败判断（重要）

**每个接口返回都有 `success` 字段**，前端统一这样处理：

```
success == true   → 显示结果（summary / items / reply）
success == false  → 弹一句 message（"智能服务暂时不可用…"），不要让页面崩
```

这样即使大模型偶尔抽风，页面也稳。

## 四、uni-app 调用示例

```js
// 智能摘要
uni.request({
  url: AI_BASE + '/api/ai/summary',
  method: 'POST',
  header: { 'Content-Type': 'application/json' },
  data: { text: bookIntro, max_length: 60 },
  success: (res) => {
    if (res.data.success) {
      this.summary = res.data.summary      // 显示摘要
    } else {
      uni.showToast({ title: res.data.message, icon: 'none' })
    }
  },
  fail: () => uni.showToast({ title: '网络异常', icon: 'none' })
})

// 智能推荐
uni.request({
  url: AI_BASE + '/api/ai/recommend',
  method: 'POST',
  header: { 'Content-Type': 'application/json' },
  data: { query: this.userInput },
  success: (res) => {
    if (res.data.success) this.recommendList = res.data.items   // 渲染书单
    else uni.showToast({ title: res.data.message, icon: 'none' })
  }
})
```

建议把这些封装成一个 `api/ai.js` 统一管理，页面里直接调。

## 五、多轮问答的会话怎么维护（用 session_id）

多轮对话的上下文由**后端自动维护**（TTL 会话，30 分钟无活动自动过期），前端**不用攒历史数组**，只需管好一个 `session_id`：

- **第一次**提问：`session_id` 不传（或传空），从返回里拿到后端生成的 `session_id`，存起来；
- **之后每轮**：带上**同一个** `session_id`，AI 就能记住上文；
- **开始新对话**（清空上下文）：换一个新的、或干脆不传，后端会新建。

```js
data() { return { sessionId: '', messages: [] } },   // 只需存一个会话 id
methods: {
  ask(userMsg) {
    this.messages.push({ from: 'user', text: userMsg })   // 先显示用户消息
    uni.request({
      url: AI_BASE + '/api/ai/chat', method: 'POST',
      header: { 'Content-Type': 'application/json' },
      data: { message: userMsg, session_id: this.sessionId },   // 首次为空
      success: (res) => {
        if (res.data.success) {
          this.sessionId = res.data.session_id   // 关键：保存后端返回的会话 id
          this.messages.push({ from: 'ai', text: res.data.reply })
        } else {
          uni.showToast({ title: res.data.message, icon: 'none' })
        }
      }
    })
  }
}
```

> 相比"前端攒 history"，这样前端只管一个 `session_id`，更省事；上下文长度、过期清理都由后端 TTL 会话处理。

## 六、联调注意

- AI 服务**已开启跨域（CORS）**，H5 直接调不会被拦。
- 接口文档/自测：打开 `AI_BASE/docs`（如 `http://localhost:8001/docs`）。
- 字段以 `schemas.py` / 本文档为准；有疑问找 AI 服务这边确认。
